**ForestOptiLM** — это имя репозитория; интерфейс продукта называется **Nocturne
Data Forge** — десктопное приложение для массовой асинхронной обработки больших
файлов и папок (в том числе корпусов на сотни тысяч токенов) через **локальные
LLM** (LM Studio REST API v1 и OpenAI-совместимые эндпоинты).

**Главная идея:** бросаешь один огромный файл или кучу больших файлов разного
формата, пишешь обычным языком, что нужно, — получаешь точный результат,
оптимизированно. Движок **не привязан к домену**: он понимает задачу, сам
выводит, *что извлекать* из каждого фрагмента, прогоняет Map-Reduce по корпусу на
локальных моделях и собирает ответ в той форме, которую подразумевает задача.
Никаких встроенных допущений о предметной области (логи, код, договоры, отчёты —
ядру всё едино).

В основе: **понимание запроса** (намерение + схема извлечения под задачу),
**Map-Reduce** с опорой на доказательства, опциональный **scout-проход** (быстрый
фильтр релевантности), **управление reasoning-моделями** (`reasoning: off` для
qwen3 / deepseek-r1), опциональная модель-**композитор** для merge/reduce,
**гибридный поиск** (BM25 + векторы) и **детерминированная агрегация** (счётчики и
группировки считаются кодом, а не выдумываются моделью).

**Целевые нагрузки (by design):** многогигабайтные логи, огромные построчные
дампы, ZIP/TAR-деревья с тысячами файлов, структурированные отчёты JSON/XML/CSV и
папки-корпуса в диапазоне **~1M+ токенов** со scout и иерархическим слиянием — а
не «один маленький PDF».

## Возможности

- **Блокноты (в стиле NotebookLM)** — постоянные именованные коллекции источников
  с **заземлённым чатом** (потоковый вывод, кликабельные ссылки `[N]` на источник
  и честный отказ *«в источниках нет ответа»*), который отвечает *только* по этому
  блокноту. Панель **Studio** генерирует учебные гайды, FAQ, таймлайны, конспекты,
  флеш-карточки и **Lint** (проверку) корпуса. Плюс **слой скомпилированных
  знаний** (паттерн *LLM-Wiki* Андрея Карпатого): модель компилирует источники в
  связанные редактируемые markdown-страницы, которые чат затем предпочитает, — это
  знание, которое **накапливается**, а не выводится заново на каждый вопрос.
  Опциональный **🎯 Точный поиск** (query-expansion + LLM-реранк) повышает recall
  и precision. См. [Блокноты](#блокноты).
- **🌐 Веб-поиск (keyless, без ключей)** — режим чата, отвечающий из *интернета*, а
  не по индексу блокнота: keyless-поиск (DuckDuckGo HTML), fetch с имитацией
  браузера + извлечение основного текста и **grounded-синтез с цитатами
  `[N]`→URL** по многим источникам (deep map-reduce). Плюс импорт статей
  **Wikipedia** и публичных репозиториев **GitLab** прямо в блокнот. Скачанные
  страницы считаются недоверенными (ограждение от prompt-injection) и не пускаются
  на внутренние/loopback-адреса (SSRF-защита).
- **Умный импорт** — распознанные экспорты чатов нормализуются в чистые,
  понятные LLM блоки `[дата] отправитель: текст` *до* индексации, а не сваливаются
  сырой разметкой. Готовые импортёры: **Telegram Desktop HTML** (`messages*.html`),
  **WhatsApp** (`_chat.txt`), JSON-экспорты **Slack** и **Discord** — с переносом
  автора у сгруппированных сообщений, пропуском служебных и маркерами `[медиа: …]`.
  Работает везде (Map-Reduce, RAG, Блокноты) и расширяется новыми форматами.
  См. [Умный импорт](#умный-импорт).
- **Несколько бэкендов** — пресеты провайдеров в один клик: LM Studio (native или
  OpenAI), **Ollama** и любой OpenAI-совместимый сервер (vLLM, llama.cpp, LocalAI);
  модели берутся из живого списка API.
  См. [Пресеты провайдеров](#пресеты-провайдеров-lm-studio--ollama--openai-совместимые).
- **Кроссплатформенные сборки** — самостоятельные приложения для **Windows, macOS
  и Linux** через CI-матрицу и пер-OS скрипты. См. [Десктоп-сборки](#десктоп-сборки-windows--macos--linux).
- **Адаптивное извлечение** — схема MAP выводится из задачи, а не зашита.
- **Конвейер Map-Reduce** с опорой на доказательства и точным слиянием.
- **Record-aware ингест** — отчёты JSON/JSONL/XML режутся по *записям* (запись
  никогда не рвётся между чанками), без пер-форматных парсеров.
- **Детерминированная агрегация** — счётчики/группировки/дедуп считаются кодом и
  передаются в отчёт как истина.
- **Гибридный поиск (RAG)** — BM25 + локальные FAISS-векторы, слитые через RRF.
- **Память прогонов** — diff корпуса с предыдущим запуском (добавлено/удалено/без изменений).
- **Scout-проход** — быстрая оценка релевантности до полного MAP (для огромных папок).
- **Пресет большого корпуса** — scout + мелкие чанки + композитор в один клик.
- **Vision** — анализ изображений мультимодальными моделями.
- **Пакетная обработка таблиц** CSV/XLSX.
- **Режим Low VRAM** — последовательная загрузка моделей для слабого железа.
- **Модель-композитор** — опциональная отдельная модель для merge/reduce/refine.
- **SQLite-кэш** для возобновления MAP с чекпойнта + кооперативный stop/resume.
- **Тёмная тема GUI** (CustomTkinter, Material Design 3).
- **Reasoning-модели** — автоопределение из каталога LM Studio; CoT отключается для MAP/REDUCE.
- **Headless CLI** — `python -m forestoptilm.cli analyze`.

## Установка

```bash
git clone https://github.com/therudywolf/ForestOptiLM.git
cd ForestOptiLM
pip install -r requirements.txt
```

### Docker

```bash
docker build -t forestoptilm .
docker run --rm -it forestoptilm
```

## Быстрый старт

### GUI

```bash
python main.py
```

### CLI (headless)

```bash
python -m forestoptilm.cli analyze ./docs -q "Сведи выводы по безопасности"
python -m forestoptilm.cli analyze ./docs -q "..." --profile large_corpus -o report.md

# Указать любой сервер (здесь Ollama) без правки конфигов
python -m forestoptilm.cli analyze ./docs -q "..." \
  --base-url http://127.0.0.1:11434 --api-mode openai -m qwen2.5 --profile ollama_local
```

Флаги: `--base-url`, `--api-key`, `--api-mode {native,openai}`, `--model/-m`,
`--profile`, `--workers`, `--composer`, `--scout-model`, `--output/-o`.

### Windows: запуск в один клик

Дважды кликните **`start.bat`** в папке проекта. Скрипт:
- создаст виртуальное окружение `.venv`, если его нет;
- пересоздаст `.venv`, если он указывает на удалённый Python;
- установит зависимости из `requirements.txt`;
- запустит приложение.

## Десктоп-сборки (Windows / macOS / Linux)

Самостоятельным сборкам не нужен Python на целевой машине. PyInstaller **не**
кросс-компилирует, поэтому каждая ОС собирается на себе — локально или в CI.

### Скачать релиз

Тегированные релизы публикуют архив под каждую ОС (собирается
[Release workflow](.github/workflows/release.yml)):

| ОС | Артефакт | Запуск |
|----|----------|--------|
| Windows | `NocturneDataForge-Windows-*.zip` | распаковать → `NocturneDataForge.exe` |
| macOS | `NocturneDataForge-macOS-*.tar.gz` | `tar xzf …` → открыть `NocturneDataForge.app` |
| Linux | `NocturneDataForge-Linux-*.tar.gz` | `tar xzf …` → `./NocturneDataForge/NocturneDataForge` (или `run.sh`) |
| Fedora | `NocturneDataForge-Fedora-*.tar.gz` | `tar xzf …` → `./NocturneDataForge/NocturneDataForge` (или `run.sh`) |

Архивы Linux/macOS — `.tar.gz`, чтобы сохранились бит исполняемости и симлинки
(обычный zip их теряет, и бинарь не запустится). Сборка **Fedora** делается внутри
контейнера `fedora:latest` для совместимости нативных библиотек; обычная сборка
**Linux** делается на Ubuntu и работает на большинстве свежих дистрибутивов.

На Linux запустите `./install-desktop.sh` из распакованной папки, чтобы добавить
**Nocturne Data Forge** (с иконкой) в меню приложений; он ставит
`~/.local/share/applications/nocturnedataforge.desktop` для текущего пользователя.

Каждый push также собирает все цели в CI ([Build workflow](.github/workflows/build.yml)),
так что артефакты можно скачать прямо из прогона Actions.

### Собрать самому

Из чекаута с установленными зависимостями (`pip install -r requirements.txt pyinstaller`):

```powershell
# Windows
pwsh -File scripts/build_exe.ps1      # или дважды кликнуть scripts\build_exe.bat
```

```bash
# macOS / Linux
bash scripts/build.sh
```

Оба предзагружают токенизатор (чтобы приложение работало **офлайн**) и запускают
PyInstaller по [`nocturne.spec`](nocturne.spec). На выходе — one-dir приложение в
`dist/NocturneDataForge/` (плюс `dist/NocturneDataForge.app` на macOS). Упаковать
для релиза: `python scripts/package_dist.py` (`.zip` на Windows, `.tar.gz` иначе).
При первом запуске приложение создаёт рядом с бинарём папку `NocturneData/` для
кэша и индексов. Артефакты сборки (`build/`, `dist/`, `.build/`) — в `.gitignore`.

### Подпись кода

Windows `.exe` несёт метаданные автора/продукта (издатель **therudywolf**, через
[`scripts/version_info.txt`](scripts/version_info.txt)), а macOS-бандл — копирайт,
но бинарники **не подписаны** — поэтому Windows SmartScreen и macOS Gatekeeper
предупредят при первом запуске (Windows: *Подробнее → Выполнить в любом случае*;
macOS: правый клик → *Открыть*, либо `xattr -d com.apple.quarantine`). Настоящая
подпись требует платного сертификата — **Authenticode** (Windows) и **Apple
Developer ID** (macOS, для нотаризации). С ними подпишите сборку через `signtool` /
`codesign`+`notarytool`; в репозитории сертификатов нет.

## Конфигурация

### Пресеты провайдеров (LM Studio / Ollama / OpenAI-совместимые)

Выберите **Провайдер (сервер LLM)** в сайдбаре — Base URL и режим API подставятся
сами, не нужно помнить порты и пути:

| Пресет | Base URL | Режим API | Примечания |
|--------|----------|-----------|------------|
| **LM Studio (REST v1)** | `http://127.0.0.1:1234` | `native` | reasoning:off для thinking-моделей, авто load/unload, контекст из метаданных |
| **LM Studio (OpenAI API)** | `http://127.0.0.1:1234` | `openai` | тот же сервер через `/v1` — если native капризничает |
| **Ollama** | `http://127.0.0.1:11434` | `openai` | OpenAI-совместимый `/v1`; без API-ключа |
| **OpenAI-совместимый** | *(введёте сами)* | `openai` | vLLM, llama.cpp server, LocalAI, … |
| **Вручную** | *(введёте сами)* | *(выберете)* | ручной режим |

Затем нажмите **Обновить модели**, чтобы получить список моделей, которые сервер
реально отдаёт, и выбрать chat / embedding / vision модель из выпадающих списков.
Пресет автоопределяется из сохранённого URL при следующем запуске.

#### Быстрый старт с Ollama

```bash
ollama serve                      # запустить сервер (:11434)
ollama pull qwen2.5               # chat-модель
ollama pull nomic-embed-text      # embedding-модель для RAG / Блокнотов
```

В приложении: Провайдер → **Ollama**, **Обновить модели**, выбрать `qwen2.5` как
LLM и `nomic-embed-text` как embedding-модель. CLI: `--base-url
http://127.0.0.1:11434 --api-mode openai`. (Эмбеддингам нужна сборка Ollama с
OpenAI-совместимым `/v1/embeddings` — есть в любом свежем релизе.)

### Файл подключения LM Studio (опционально)

Скопируйте шаблон и впишите свои значения:

```bash
cp config/lmstudio.example.json .local/lmstudio.json
```

```json
{
  "base_url": "http://127.0.0.1:1234",
  "api_key": "sk-lm-xxxxxxxx:yyyyyyyy",
  "timeout": 600,
  "default_model": "",
  "api_mode": "native"
}
```

Папка `.local/` — в `.gitignore`, секреты в репозиторий не попадают.

Альтернатива: переменная окружения `NOCTURNE_LMSTUDIO_CONFIG` с полным путём к
JSON-конфигу.

### Значения по умолчанию

Если локального конфига нет, используется `http://127.0.0.1:1234` без API-ключа.
Создайте `.local/lmstudio.json` из шаблона (и никогда его не коммитьте).

### LM Studio 0.4+ REST API v1

При включённой **Server Settings → Require authentication** создайте токен в
**Manage Tokens** (`sk-lm-...`) и впишите в `.local/lmstudio.json`. Приложение использует:

| Эндпоинт | Назначение |
|----------|------------|
| `GET /api/v1/models` | Список моделей |
| `POST /api/v1/chat` | MAP / REDUCE / scout |
| `POST /api/v1/models/load` | Загрузка модели в Low VRAM (`context_length` поддерживается) |
| `POST /api/v1/models/unload` | Выгрузка инстанса |

По умолчанию режим native (`api_mode: native` в локальном конфиге или GUI).

### Режим API

В GUI доступен выбор:

| Режим    | Эндпоинты |
|----------|-----------|
| `native` | LM Studio REST (`/api/v1/*`) |
| `openai` | OpenAI-совместимый (`/v1/*`) |

Поле URL принимает корневые URL LM Studio и скопированные API-адреса:
`http://127.0.0.1:1234`, `http://127.0.0.1:1234/v1`, `http://127.0.0.1:1234/api/v1`.

## Бюджет токенов

Контекст считывается из метаданных LM Studio автоматически:

1. `loaded_context_length` (рантайм)
2. `context_length` / `max_context_length` из метаданных
3. Фолбэк: `8096`

```
response_reserve = clamp(0.2 * effective_context, 1024, 4096)
chunk_size = effective_context - system_prompt - user_query - reserve
```

Защитный лимит (`NOCTURNE_MAX_CHUNK_TOKENS`, по умолчанию `6000`, `0` = выкл)
ограничивает максимальный размер чанка.

## Большие файлы и архивы

| Вход | Поведение |
|------|-----------|
| **Текст/код/лог ≥ 50 MiB** (`NOCTURNE_STREAMING_FILE_BYTES`) | Построчный стриминг в MAP-чанки — файл **не** грузится в RAM целиком |
| **ZIP / TAR / `.tar.gz`** | Распаковка во временную папку; каждый файл сохраняет свой **путь** (как при выборе папки) |
| **Папка** | Каждый поддерживаемый файл → чанки → один Map-Reduce (рекомендуется scout) |

Лимиты практические: каждый MAP/SCOUT-чанк всё равно зовёт вашу локальную LLM.
Файл на **триллион строк** поддерживается в смысле **потокового ввода**; общее
время зависит от пропускной способности LM Studio, порога scout и железа. Для
огромных корпусов используйте **scout** + профиль `large_corpus`.

| Переменная | Умолчание | Смысл |
|------------|-----------|-------|
| `NOCTURNE_STREAMING_FILE_BYTES` | `52428800` (50 MiB) | Стримить файлы от этого размера (`0` = всегда грузить целиком) |
| `NOCTURNE_MAX_ARCHIVE_BYTES` | `8589934592` (8 GiB) | Отказ распаковывать архивы крупнее (по сжатому) |
| `NOCTURNE_MAX_UNCOMPRESSED_BYTES` | `8589934592` (8 GiB) | Отказ, если **распакованный** размер превышает (защита от zip/tar-бомб; `0` = выкл) |
| `NOCTURNE_RECORD_AWARE` | `1` | Структуро-сохраняющий ингест JSON/JSONL/XML — записи не рвутся между чанками (`0` = старый text/table путь) |
| `NOCTURNE_RUN_MEMORY` | `1` | Сохранять извлечённое за прогон и показывать diff с предыдущим (`0` = выкл; алиас: `NOCTURNE_FINDINGS_MEMORY`) |
| `NOCTURNE_RECORD_AWARE_MAX_BYTES` | `209715200` (200 MiB) | Выше этого большой JSON/XML падает на стриминг текстом (JSONL всё равно по записям) |
| `NOCTURNE_MAX_CHUNKS_IN_RAM` | `12000` | Сбрасывать MAP-чанки в SQLite сверх этого количества |
| `NOCTURNE_MAP_BATCH_SIZE` | `workers × 4` | Размер батча параллелизма MAP |
| `NOCTURNE_MAP_NORMALIZE_SPILL` | `2500` | Сброс нормализованного MAP-JSON в SQLite до merge (`0` = всё в RAM) |
| `NOCTURNE_DUAL_MAP_RESOLVE` | `0` | При `1` и пуле из 2+ инстансов — MAP на обоих и merge через `conflict_resolve` |

### Профили оптимизации (run profiles)

[`config/run_profiles.yaml`](config/run_profiles.yaml) собирает настройки движка
(scout-фильтр, размер чанка, воркеры, composer-merge, low-VRAM) в именованные
пресеты, применяемые `--profile <name>` (CLI) или кнопками GUI:

| Профиль | Для чего | Особенности |
|---------|----------|-------------|
| `quick_scan` | быстрый первый проход | scout вкл, мелкие чанки |
| `balanced` | средние корпуса | без scout, чанки 5k |
| `precise` | максимальная точность | мелкие чанки, composer-merge |
| `deep_audit` | тщательный разбор | без scout, крупные чанки |
| `large_corpus` | 1M+ токенов | scout + composer + low-VRAM |
| `ollama_local` | Ollama / малый контекст | scout, чанки 3k |

Сам промпт извлечения **выводится из запроса** (намерение + схема под задачу),
поэтому пресеты крутят *пропускную способность и контекст*, а не формулировку —
она подстраивается сама. Размер контекста читается с сервера автоматически;
ограничить можно `NOCTURNE_MAX_CHUNK_TOKENS` или панелью **Дополнительно**.

### GUI (без ручной настройки)

- **Быстро / Глубоко / 1M+** — профили в один клик.
- **Оценка (dry-run)** — оценка чанков/файлов и грубый ETA без вызовов LLM.
- **История** — недавние прогоны из БД метрик.
- **Продолжить** — возобновить прерванный MAP из SQLite-кэша. Resume совпадает по
  файлу/папке + запросу **и** параметрам прогона (размер чанка, MAP-модель,
  composer): смена модели или профиля запускает свежий прогон, а не тянет старые чанки.
- Большой файл / ZIP / папка → **авто** пресет large_corpus при старте.

Чекпойнты MAP и метаданные заданий живут в `.nocturne_cache/` (или
`NOCTURNE_CACHE_DIR`). Указатель последнего незавершённого задания — в
`.local/last_job.json` рядом с конфигом LM Studio.

## Конвейер

1. **Понимание** — запрос разбирается в `QueryPlan` (намерение, ключевые термины,
   ось группировки, стиль вывода) и **схему извлечения под задачу** (что тянуть из
   каждого фрагмента). Без привязки к домену — severity по безопасности это лишь
   одно опциональное поле, используется только когда задача про проблемы/качество/риск.
2. **MAP** — каждый чанк даёт структурированный JSON: извлечённые элементы с
   `type`, содержимым, опциональными полями под задачу и `evidence_refs`
   (`file`, `chunk`, `quote`) для заземления.
3. **Merge** — иерархическое детерминированное слияние JSON для больших корпусов.
4. **Aggregate** — счётчики / группировка / дедуп считаются **кодом** (по ключу из
   QueryPlan) и передаются в REDUCE как истина, поэтому числа верны на любом объёме.
5. **REDUCE** — финальный Markdown-ответ; контракт вывода под намерение формирует
   форму (таблица / ранжированный список / сравнение / сводка / отчёт).
6. **Refine** — второй проход, если ответ слишком короткий или неполный.
7. **Validation** — предупреждения при коротком выводе, отсутствии структуры или
   низкой плотности доказательств.

Элементы, заявленные с высокой важностью без доказательств `file + quote`,
автоматически понижаются — выводы должны опираться на источник.

Извлечённые элементы дедуплицируются (по ключу из запроса) и ограничиваются на
чанк и на уровень слияния — см. `NOCTURNE_MAX_FINDINGS_PER_CHUNK` и
`NOCTURNE_MERGE_FINDINGS_CAP` для очень больших корпусов.

## RAG

1. Выберите embedding-модель в GUI.
2. Выберите файл или папку, откройте вкладку **RAG**, нажмите **Build Index**.
3. Введите вопрос и нажмите **Ask**.

Поиск **гибридный**: плотные векторы (FAISS) для семантики + BM25 для точных
токенов (`CVE-2024-3094`, хосты, `pkg@version`), слитые через Reciprocal Rank
Fusion. Если embedding-модель/сервер недоступны, работает чистый BM25 — точные
запросы не зависят от эмбеддера.

## Умный импорт

Сырые экспорты индексируются плохо: Telegram-дамп через наивный `get_text()`
превращается в кашу из навигации, дат, реакций и разметки. **Умный импорт**
распознаёт известные форматы экспорта и переписывает их в чистый по-сообщениям
текст *до* чанкинга — в той форме, что люди вручную готовят для NotebookLM.

- **Telegram Desktop HTML** — укажите приложению (или источнику блокнота) на файл
  `messages*.html` или папку экспорта. Каждое сообщение становится
  `[<дата+таймзона>] <отправитель>:\n<текст>`; сгруппированные сообщения наследуют
  автора предыдущего (Telegram его опускает), служебные `message service`
  отбрасываются, фото/файлы/стикеры получают маркер `[медиа: …]` вместо исчезновения.
  Сохраняется структура треда: `[В ответ на …]` для ответов и `[переслано от: …]`
  для пересланного.
- **WhatsApp** (`_chat.txt`), JSON-экспорты **Slack** и **Discord** — та же чистая
  нормализация `[дата] отправитель: текст` (импортёры `whatsapp_txt`, `slack_json`,
  `discord_json` в `smart_import.py`).
- **Generic chat-логи** (`имя: сообщение`, без таймстампов) — консервативный
  эвристический импортёр (`generic_chatlog` ) берёт простые диалоги, но игнорирует
  конфиги `ключ: значение` и прозу; запускается последним, чтобы специфичные выигрывали.

Распознавание и конвертация происходят внутри
[`file_extractors.extract_content`](file_extractors.py), так что умный импорт
работает и в Map-Reduce, и в RAG, и в Блокнотах — без отдельной кнопки.
Нераспознанное падает на обычный путь `html→text`. Новый формат — это один класс
в [`smart_import.py`](smart_import.py).

## Блокноты

**Блокнот** — постоянная именованная коллекция источников, с которой можно
общаться и которую можно изучать — локальный взгляд на NotebookLM. Каждый блокнот
самодостаточная папка; всё остаётся на вашей машине.

Вкладка **Блокноты** открывается на **архиве исследований** — галерее карточек
(обложка, описание, число источников/фрагментов, статус индекса, поиск), к которым
возвращаешься, переоткрываешь, переименовываешь, описываешь и которыми управляешь.
Открытие карточки переключает в **рабочее пространство** из трёх колонок —
**Источники · Чат · Studio** — как в NotebookLM.

1. В архиве нажмите **＋ Новое исследование** (затем **Изменить** — задать
   описание/иконку; цвет обложки автоматический).
2. Добавьте источники: **Файл** / **Папка** (индексируются на месте — огромные
   дампы не копируются) или **URL** (текст страницы скачивается в блокнот).
3. Нажмите **🔨 Построить индекс** — соберёт собственный гибридный индекс блокнота
   (BM25 + FAISS) выбранной в сайдбаре embedding-моделью.
4. **Чат:** спрашивайте обычным языком. Ответы **заземлены** — поиск идёт *только*
   по этому блокноту, каждое утверждение несёт ссылку `[N]`, клик по чипу-цитате
   показывает точный фрагмент с кнопкой открыть источник. Если ответа в источниках
   нет — ассистент честно скажет, а не выдумает (и всё равно покажет найденные
   фрагменты-кандидаты). Ответ **стримится** по токенам, **⏹ Стоп** прерывает его
   на лету, а **📌 В знания** подшивает хороший ответ в вики постоянной страницей.
   - **🎯 Точный поиск** (тумблер, по умолчанию выкл — добавляет ~2 вызова модели):
     LLM переписывает вопрос в 1-2 перефразировки (выше recall), а listwise
     LLM-реранкер переупорядочивает кандидатов по реальной релевантности (выше
     precision). По мотивам рекомендаций LLM-Wiki Карпатого и `qmd`.
5. **Studio:** один клик генерирует **учебный гайд**, **FAQ**, **таймлайн**,
   **конспект** или **флеш-карточки** по корпусу; результаты сохраняются заметками.
   **🔍 Проверить блокнот** (Lint) проверяет корпус на противоречия,
   устаревшие/дублирующиеся факты и пробелы.
6. **📚 Скомпилировать знания (вики):** вместо вывода всего из сырых фрагментов на
   каждый вопрос (амнезийный RAG) модель один раз компилирует корпус в набор
   связанных **редактируемых** markdown-страниц — **Обзор / Сущности / Понятия /
   Глоссарий** — в `wiki/`, плюс человекочитаемый каталог `index.md` и
   append-only журнал `log.md`. Вики индексируется отдельно, и заземлённый чат
   **предпочитает это скомпилированное знание** (цитируя читаемую вики-страницу), а
   сырые фрагменты идут для деталей. Это паттерн *LLM-Wiki* Андрея Карпатого —
   знание, которое **накапливается**, а не открывается заново каждый раз.
   Опционально и недеструктивно (сырьё не трогается); папка `wiki/` чисто
   открывается в Obsidian.

**Схема** блокнота (в **Изменить**: короткое свободное описание домена — что за
данные, какие сущности важны) направляет и system-prompt чата, и компиляцию вики.
Картинки **внутри `.docx`/`.pdf`** (диаграммы, схемы) описываются vision-моделью
на индексации при включённом **🖼 Описывать картинки** — и «что на этой диаграмме»
становится отвечаемым.

Блокноты живут в `NocturneData/notebooks/` рядом с упакованным `.exe`, или в
`.local/notebooks/` при запуске из исходников (переопределяется
`NOCTURNE_NOTEBOOKS_DIR`). Структура блокнота: `notebook.json` (метаданные, вкл.
опциональную `schema`), `index/` (FAISS + BM25 по сырью), `sources/`
(URL/производный текст), `notes/` (вывод Studio), `wiki/` (страницы знаний +
`index.md`/`log.md` + `answers/` из 📌 В знания), `wiki_index/` (отдельный индекс по
вики), `chat.jsonl` (история чата). Добавление источника пересобирает индекс (FAISS
неизменяем). Аудио (`.mp3/.wav/.m4a/.ogg/.flac/.opus/.aac/.wma`) транскрибируется
локально опциональной зависимостью `faster-whisper` и становится текстовым
источником; только TTS / аудио-«обзоры» сознательно вне scope.

## Интерфейс

UI следует **Material Design 3** (тёмная тема): единый модуль токенов
[`md3.py`](md3.py) задаёт цветовые роли (seed indigo-violet), типографическую
шкалу, токены формы/скругления и spacing по сетке 4dp, а `md3.apply()` оформляет
сразу все виджеты CustomTkinter — скруглённые углы, тональную лестницу
поверхностей, аутентичную иерархию кнопок (filled / tonal / outlined / text),
тональные чат-пузыри и карточки с elevation. Менять акцент, радиусы или
типографику можно в одном месте.

Изменения UI проверяются **headless** (без реального дисплея) через
[`tools/ui_shot.sh`](tools/ui_shot.sh) + [`tools/ui_preview.py`](tools/ui_preview.py):
они рендерят ключевые экраны под Xvfb (например, в WSL) и снимают скриншот;
превью импортирует те же `md3`-токены, поэтому снимок честно отражает приложение.

## Диагностика

- **Модели дублируются или не выгружаются** — включите режим *Low VRAM
  Sequential*, переключите режим API и повторите.
- **Частые ошибки 400** — проверьте совместимость режима API; смотрите вкладку
  Логи и классификатор `400` (`payload_mismatch`, `unsupported_option`, `context_limit`).
- **Сбросить настройки GUI** — удалите `.local/ui_runtime.json` и перезапустите.
